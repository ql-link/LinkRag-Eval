# LinkRag-Eval 质检模块全解:一个 RAG 系统是如何"自我体检"的

> 本文是概念说明，不作为当前进度或接口真值源。当前状态见 [CURRENT_STATUS.md](../CURRENT_STATUS.md)，权威实现边界见 [解耦架构](../architecture/decoupling-plan.md)。

## 引子:为什么 RAG 需要一套独立的质检系统

一个生产级的 RAG(检索增强生成)系统,链路很长:文档解析清洗、切分入库、多路召回、重排、拼上下文生成答案。每一环都可能出问题——PDF 表格解析乱了、召回参数调歪了、重排模型没起作用、生成的答案看似流畅实则答非所问。如果没有量化的质检手段,这些问题只能靠人工抽查,既不可持续,也无法在每次改动(换模型、调参数、升级 parser)后快速确认"有没有变差"。

`LinkRag-Eval` 就是为解决这个问题而生的独立评测/质检项目。它从生产 RAG 仓库 `toLink-Rag` 中剥离出来,不与生产共用存储、不共用模型解析链路,只通过"产物级纯函数"复用生产的计算能力——自己负责入库、检索、算分。这篇文章按照"总体架构 → 分阶段质检内容 → 指标体系与选型理由 → 黄金测试集的构建与权威性"的顺序,把整个质检模块讲清楚。

---

## 一、总体架构:为什么要与生产解耦

在剥离之前,评测代码直接嵌在生产仓库里,存在三个问题:

第一,评测代码直接 import 生产里"算法与写库绑死"的三个 pipeline(`EsIndexingPipeline`、`SparseIndexingPipeline`、`VectorStoragePipeline`)以及生产的写库 ORM,导致评测环境的任何一次改动都可能牵连生产代码路径。第二,评测与生产共用同一个 Qdrant collection,生产端的日常维护(下线 ES、迁移向量 schema)会直接误伤评测数据。第三,评测被生产的"按用户解析模型配置"逻辑(`aresolve_user_model → llm_user_config`)拖回共享 MySQL 连接池,产生资源竞争。

解耦方案的核心原则是:评测系统只通过"产物级纯函数"复用生产的计算能力,自己负责入库、检索、算分,不共用任何有状态资源。落地为四条具体约束:

- 独立 Python 包 `linkrag_eval`,采用 src-layout,但包名刻意不叫 `src`——因为生产仓库 toLink-Rag 自身就是一个叫 `src` 的顶层包,两个同名顶层包会在 `sys.path` 里互相遮蔽,而评测恰恰需要同时 import 两边。
- 独立 MySQL 库 `tolink_rag_eval_db`,与生产同一台服务器、但库级隔离,复用账号只换库名。
- Qdrant 使用独立的 `eval` 前缀 collection,同一个 host,但 `EvalVectorStore` 在构造时会强制断言 collection 前缀含有 `eval`,不满足就直接抛异常拒绝运行——这是防止误写生产数据的最后一道防线。
- 评测专用的 LLM 判官客户端(`judge/eval_llm.py` 里的 `EvalChatClient`)完全独立于生产的模型解析链路。设计文档说得很直接:判官是"测量仪器",不是"被测对象",如果测量仪器绑定在被测系统的配置解析链上,生产改一次模型解析方式就会拖垮评测,还会和生产共享连接池、给生产的用量遥测灌入噪声。

哪些生产代码允许被评测 import,划出了一条很窄的白名单:纯计算函数(文本切分、BM25 分词)、Qdrant 的读写原语、以及"被测对象"本身(召回管线、检索器)。凡是"写 pipeline"、生产 ORM、全栈解析任务、对象存储工厂、消息队列、per-user 配置解析,一律禁止 import,由 `.importlinter` 规则和 CI 里的边界测试机器强制执行,不靠人自觉。

这套边界背后还有一个协议化的抽象:`ProductComputer`。它把"文本切分""生成 dense 向量""生成 sparse 向量""BM25 分词"这四个纯计算能力收口成统一接口,唯一实现类是 `rag_adapter.py`。评测的其余代码只依赖这个协议,不直接碰生产内部实现。这样做的好处是,生产 RAG 包升级、内部函数签名变化时,只需要改这一个适配器文件,其余评测代码不受影响;配套的契约测试会在生产包升级时自动校验每个方法的输出形状是否漂移。

## 二、质检分几个阶段:从"数据清洗"到"最终生成"

质检模块把一个 RAG 系统拆成五个纵向质检层:**数据清洗(CLEANING)→ 检索召回(RETRIEVAL)→ 重排(RERANK)→ 生成(GENERATION)→ 正确性(CORRECTNESS)**。工程上按 Phase 0 到 Phase 3 分批落地,原则是先把与"评什么"无关的地基打好,再打通检索层形成第一个可交付的闭环,然后依次补齐重排层和生成层——越往后越依赖主观判断,也就越晚啃。

### Phase 0:地基层

这一阶段不接触任何外部系统,只做两件事:定义抽象契约(`Evaluable`/`Metric`/`Judge`/`Dataset`/`Sample`)和定义统一的数据结构。其中最关键的设计是 `StageOutput`——用一个结构同时承载检索、重排、生成三层的产出(排序结果、对照序列、生成答案、引用上下文),让"指标怎么算"和"具体是哪一层产出的"彻底解耦。这里有一条铁律:指标计算只认归一化后的字段(排序列表、答案文本、上下文),不允许依赖某个环节的原始返回值,原始返回值只能用来辅助诊断,不能进入打分逻辑。

这一阶段也确立了"冻结语料"的概念:评测使用的租户号段(`EVAL_USER_ID=990001`)只是一个路由/分区常量,不代表真实用户身份,评测语料一次性灌入后就冻结为只读,后续如果要重新分块或换存储,必须先重新校验数据完整性,不能带着假设直接开跑。

### Phase 0.5:数据清洗质检——离入库最近的一道关口

文档解析(Word/PDF/HTML 转 Markdown 再入库)这一步的质量,过去只能通过下游召回效果"倒推"是否有问题,没有直接的量化手段。这一阶段设计了一个巧妙的方法——**Markdown 回环测试(round-trip)**:

拿一批已知干净的真实中文 Markdown 语料作为"参考真值",把它渲染成 Word/PDF/HTML 等格式,再用生产的 parser 把这些渲染件重新解析清洗回 Markdown,最后拿"解析产出"和"参考真值"做比对。这样做的好处是,参考真值不需要人工书写、不需要 LLM 编造,天然贴近真实文档结构,也避免了"用清洗流程的结果反过来验证清洗流程"这种循环论证。对于开源真实语料覆盖不到的结构边界(比如极端的表格嵌套),再用少量 LLM 合成文档补充。

这一阶段的质检对象包括:文本是否完整、噪声是否混入、标题层级是否识别正确、表格是否保真(区分"仍是 Markdown 表格""被转成图片""被转成结构化 JSON"三种清洗产物,分别用不同方式打分)、图片是否被正确识别及其上下文位置是否正确、列表与顺序是否保真、同一文档多次清洗结果是否稳定(用于发现使用了非确定性后端的解析器),以及单文档清洗的耗时——这个耗时被列为一等指标,因为它直接关系到后端选型的成本权衡。所有指标都按"文档格式 × PDF 解析后端"分桶统计,不同技术路线不混在一起看平均数。

需要说明的是,这一步测的是"渲染 + 清洗"两步叠加后的保真度,渲染本身也会影响标题呈现方式,这是文档里明确写出的边界,不回避;而"分块(chunk)质量"则完全不在这一模块的范围内,由检索层间接体现。

### Phase 1:检索召回质检——第一个可交付的闭环

这是整个质检模块里最先落地、也是当前唯一完整跑通全流程的一层。它遵循两条原则:第一,指标计算函数是零外部依赖的纯函数,可以直接进 CI 单元测试;召回执行本身需要连接真实的向量库和模型服务,只能人工或定时触发,不进 CI。第二,召回口径必须与生产完全一致——直接复用生产的召回管线装配逻辑,而不是评测自己另起炉灶拼三路召回,否则 top_k、融合策略、供应商配置一旦和线上出现偏差,测出来的数字就没有意义。

这一层的输入是黄金测试集(下一节详细讲),每条样本包含查询语句、期望命中的 chunk 或文档 id、问题类型(关键词直问/同义改写/长尾细节/跨文档多跳)等。加载黄金集时有一步"前置校验"(precheck):逐条确认样本里声明的期望命中项,在当前的评测语料里确实存在。如果对不上,这一轮直接判定无效并逐条列出问题,而不是硬跑出一个可能失真的分数——这体现了整个项目最看重的一条原则:数据可复现,优先于指标好看。

### Phase 2:重排质检——衡量"重排到底有没有起效"(目前仅设计文档,代码未落地)

重排层评测要回答的核心问题不是"重排后的指标是多少",而是"相比不重排(仅用融合排序)相比,重排到底带来了多少提升"。因此设计上要求同一轮评测里,基于完全相同的候选集合(先统一把查不到正文的候选过滤掉,保证两条序列可比),同时产出"重排生效"和"重排降级"两个版本的排序结果,再计算两者在 NDCG、MRR 上的差值。这里有一个容易被忽略的陷阱:如果某些样本因为各种原因触发了降级(重排没有真正生效),这些样本的差值天然是零,如果不把"重排实际生效的样本占比"单独报出来,大量降级样本会把整体提升幅度稀释成一个虚假的"重排没什么用"的结论。所以报告里必须同时给出"全体样本的提升幅度"和"仅统计重排真正生效样本的提升幅度"两个数字。

### Phase 3:生成与正确性质检——最主观、也最晚落地的一层(目前仅设计文档,代码未落地)

这一层要回答两类不同的问题:答案是否忠实于检索到的上下文、是否切题(不需要标准答案,可以直接对现有回答打分);答案是否正确(需要黄金答案做对照)。由于"忠实度""相关性"本质上是语义判断,没法用数学公式精确定义,只能引入 LLM 作为"判官"打分,首版直接采用业界已经验证过的 RAGAS 框架的 RAG Triad 评测思路,而不是自己从零设计打分标准。

这一层有一条不可妥协的纪律:参与同一轮评测的生成模型、门禁复核模型、判官模型、被测对话模型,两两之间不能是同一个模型,系统会显式校验并在撞名时告警,目的是防止"自己出题自己判卷"式的自评偏置。但文档也非常诚实地承认,这条纪律只能防止"同名模型"这种最明显的偏置,防不住"虽然模型不同、但同源于相近训练数据和相近判断倾向"这种更隐蔽的偏置——所以生成层在合成数据集上跑出来的绝对分数天然偏乐观,只能用于同一套配置换了参数或模型之后的相对趋势追踪,不能当作绝对质量的权威结论。

## 三、指标体系:为什么选这些指标,而不是别的

质检模块的指标严格按"评的是哪一层"分组,彼此不混用、也不共享同一套计算逻辑。总的路线选择是:凡是能用确定性公式衡量、不需要主观判断的环节(数据清洗、检索),一律用零外部依赖、可以直接进 CI 门禁的"引用式"(reference-based)指标;重排层复用检索层同一套公式、只是多算一个"相对基线的增量";生成与正确性层的"忠实度""相关性"这类语义判断无法用公式表达,只能引入 LLM 判官。

下面按四层分别说明。文中的阶段状态是成文时快照：数据清洗层和检索层已有完整代码；重排与生成/正确性层的早期设计归档在 [phase2_rerank_design.md](../archive/design-v1/phase2_rerank_design.md) 和 [phase3_generation_design.md](../archive/design-v1/phase3_generation_design.md)。后续进展统一以当前状态页为准。

### 3.1 数据清洗层(CLEANING)指标——回答"入库前的内容有没有失真"

清洗层指标只用在 Phase 0.5,评的是"parser 把 Word/PDF/HTML 转成 Markdown"这一步的保真度,和检索排序完全无关,计算逻辑也完全独立于检索层。计算方式是拿"参考真值 Markdown"和"parser 实际清洗产出的 Markdown"各自解析成标题、段落、表格、列表、图片等结构化块序列,归一化(统一大小写、去标点、全角半角转换)之后逐类比对,全部是不依赖任何模型、不读像素的纯文本比对函数,可以直接进单元测试。具体包括:

- **文本完整性(recall)**:参考文本里的 token 有多少比例被清洗产出保留下来,抓的是丢字、漏段、漏页这类问题。
- **文本噪声(precision)**:清洗产出里有多少比例是参考文本里没有的多余内容,抓的是乱码、页眉页脚、水印混入这类问题。
- **标题识别完整率**:原文里的标题,有多少比例被正确识别出来,专门抓"标题被当成正文漏识别"的问题;配套还有一个反向的**误识别率**,抓"正文或页眉被误判成标题"的问题。这一项对 PDF 格式最敏感,因为大多数 PDF 没有标题树元数据,parser 只能靠字号、加粗、编号这些版面线索去猜哪些文字是标题、是几级标题,DOCX/HTML 这类自带标题标签的格式则作为对照基线。
- **标题层级一致率**:在已经识别出来的标题里,层级(几级标题)和原文是否一致,分"完全一致"和"整体结构同构、允许统一偏移"两个口径分别报,专门抓"h2 被识别成 h1 或 h3"这类层级错乱。
- **表格识别**:先判断 parser 把这张表清洗成了三种形态中的哪一种——仍是 Markdown 表格、被转成了图片、还是被转成了 JSON,再按形态分别评分:仍是 Markdown 表格的,按单元格位置和内容算对应关系的 F1 值;转成图片的,只能判断表格在文档里的位置对不对、有没有被完整框进图片里,内容层面难以逐格比对;转成 JSON 的,把 JSON 还原成"第几行第几列是什么值"的三元组集合,和原表的三元组集合做对齐,同样算一个对应关系 F1。三种形态都要先判断表格在文档中的相对位置是否正确,这是共同的底线要求。
- **图片识别完整率与上下文位置正确率**:原文里的图片有多少比例被正确保留成了 Markdown 图片引用(抓图片丢失),以及每张图片的前一个文本块和后一个文本块是否和参考里的位置一致(抓图片插错位置、漂移错位),因为图片本身的像素内容难以比对,所以重点看位置锚点而不是看图片内容。
- **列表保真**:列表项数量和嵌套层级是否还原正确。
- **顺序保真**:各个结构块之间的相对顺序是否和原文一致,抓段落乱序、跨栏内容错排的问题。
- **稳定性**:同一份输入反复清洗多次,输出是否一致,专门用来发现使用了非确定性后端(比如某些基于视觉大模型的 PDF 解析后端)的解析器,这类后端同样的输入可能每次输出都不完全一样。
- **数据清洗时间**:单篇文档清洗的实际耗时,这项被列为一等指标,直接服务于"要不要换一个更贵但更准的解析后端"这类成本权衡决策,不是随手记录的辅助信息。

这些指标全部按照"文档格式 × PDF 解析后端"两个维度分桶统计,不会把不同技术路线的数字混在一起算平均——因为不同格式、不同后端天然难度不同,混在一起看均值没有意义,只有分桶之后同格式同后端之间的对比才有决策价值。

需要特别说明为什么清洗层这套指标是"引用式"而不需要 LLM 判官:参考真值就是最初那份干净的标准 Markdown 本身,不需要人工标注、不需要模型打分,清洗产出和参考真值之间的差距就是清洗失真的直接度量,这也是为什么整个清洗质检子模块能做到"零人工"——真值不是判断出来的,而是天然存在的。

### 3.2 检索层指标(已实现)——回答"广撒网召回,有没有漏掉标准答案"

检索层指标同样是引用式的,但参考真值不再是"另一份文本",而是黄金测试集给出的"这道题标准答案对应哪些文本块或文档"这个标注集合,和清洗层比对的对象、比对的方式完全不同,不能混为一谈。代码落在 `src/linkrag_eval/metrics/retrieval.py`,是纯函数、零 IO,已经反复在实证报告里跑出真实数字。设标准答案集合为 R,排序结果为按名次排好的候选列表,具体指标含义如下:

**Recall@K(召回率)**:前 K 个结果里命中了多少比例的标准答案。这是检索层最主要看的指标,回答"广撒网式的召回,有没有把标准答案漏掉"这个最基本的问题。

这里有必要把计算口径讲精确,因为它是逐题(逐 query)算、再取平均的,不是笼统地看一个总占比。设某一道题的标准答案集合是 R,前 K 个候选结果是 topK,这一道题的 `Recall@K = |topK ∩ R| / |R|`。举例说,如果这道题标注了 3 个标准答案,而 Recall@10 的前十个候选里命中了其中 2 个,那么这道题的 Recall@10 就是 2/3 ≈ 0.667,不是命中数直接除以 10、也不是除以数据集里标准答案的总数。最终报告里看到的"整体 Recall@10",是把黄金测试集里每一道题各自算出来的这个比值,再取平均。所以哪怕不同题目标注的标准答案数量不一样(有的题只标 1 个,有的题标 3 个甚至更多),每道题都是各自除以各自的分母,分母的差异不会互相影响,加总平均之后就是最终报出来的那一个数字。

标准答案数量本身也不是固定 1 个:开源数据集转换代码(`golden/opensource/convert.py`)里,一道题的正例段落可以映射到多个文档 id,`expected_doc_ids` 完全可能是一个多元素集合;自有语料反向合成里的"跨文档多跳"(`cross_doc`)问题类型,`expected_chunk_ids` 也会包含来自多个相邻文本块的 id。只是关键词直问、同义改写这类由单个文本块反向生成的问题,天然只对应 1 个标准答案,数量上更常见一些,但不能理解成"系统固定只有 1 个标准答案"。

**Hit Rate@K(命中率)**:前 K 个结果里只要命中一个标准答案就算 1,不命中算 0,不关心命中了几个、排在第几。这个指标最直观,适合讲给不熟悉检索技术的读者听:"十次里有几次真的找到了"。它和 Recall@K 的区别正好对应上面的例子:同样是标注 3 个标准答案、命中 2 个的情况,Recall@10 记 2/3,但 Hit Rate@10 只要命中了至少 1 个就直接记 1,不区分命中 1 个还是命中 3 个。

**Precision@K(准确率)**:前 K 个结果里有多少比例是真正相关的,计算是 `|topK ∩ R| / K`。因为不少题目标注的标准答案数量少于 K(比如只标 1 个,K 却取 10),分母 K 天然偏大,这个指标数值通常偏低,不适合单独拿绝对值下结论,需要结合 Recall 一起看。

接着用同一个例子算一遍:还是那道标了 3 个标准答案的题,Recall@10 的前十个候选里命中了其中 2 个,那么这道题的 `Precision@10 = 2/10 = 0.2`——分母固定是 10(K 本身),不管这道题标了几个标准答案。再看另一种更常见的情况:一道题只标了 1 个标准答案,前十个候选里刚好命中了这唯一的 1 个,`Recall@10 = 1/1 = 1.0`(标准答案全被找到,召回意义上是满分),但 `Precision@10 = 1/10 = 0.1`(十个位置里九个都是"陪跑"的候选,自然不相关)。这就是为什么 Precision@K 在这套评测体系里数值普遍偏低、且不能单独拿绝对值下结论——它反映的更多是"K 定得多宽",而不是"检索有多不准";真正要看检索准不准,还是回到 Recall@K 和 Hit Rate@K。

**MRR(平均倒数排名)**:第一个命中项排在第几位,排得越靠前分数越高。这个指标对排序位置敏感,检索层也会看,但它真正的主场是下一节的重排层。

**NDCG@K(归一化折损累计增益)**:同时考虑"有没有命中"和"命中项排在第几位"。这里有二值相关(命中/不命中)和分级相关(相关程度分成多档,比如非常相关/比较相关/不太相关)两种口径,两者数值不可直接比较,也不能混在一张图上连线。

**MAP(平均准确率均值)**:综合考虑排序中每个相关项出现位置的准确率,分母用"标准答案数与候选数取较小值",避免了当标准答案数量多于候选返回数量时,指标虚高的问题。

**SourceOverlap(多路召回重叠度)**:统计 dense(语义向量)、sparse(稀疏向量)、BM25 三路召回各自独有命中和共同命中的占比,用来判断三路是否存在冗余、是否值得继续维护三路并行的架构。这个指标的巧妙之处在于,它不需要额外单独调用每一路召回,直接从融合结果里各路打分是否非空推导出来,既省了一次调用,又保证和真正参与融合的输入完全一致。

检索层的口径约定是:主看 Recall@K,NDCG@K、MRR 等排序敏感指标作为辅助参考(它们真正被当作"主看指标"是在重排层)。

关于门槛值,比如项目里反复提到的 `recall@10 ≈ 0.901`,需要澄清它的真实含义:这不是"质量及格线",而是重构验证的门槛——每次对底层基础设施做改动(换存储方式、换模型供应商装配方式、抽象计算接口等),都要拿这个数字做回归检查,确认改动没有意外改变系统行为,而不是在宣称"0.901 就是及格分数"。实际运行中这个数字出现过明显波动:某次复跑降到 0.8919,排查发现是两条样本被稀疏检索的低分噪声在融合环节挤出了前十,后续通过网格搜索调整稀疏检索的过滤阈值后修复,最终固化的基线跑到了 `recall@10=0.9745`。这也印证了指标背后一定要配合归因分析,而不是只看数字涨跌。

更进一步,项目里对"什么才算真正的性能回归"也定了统计纪律:不能拿裸阈值(比如"跌超过 2 个百分点就算回归")直接下结论,因为黄金测试集分桶后往往每桶只有一二十条样本,裸阈值几乎必然会被随机波动误触发。正确做法是先做"噪声地板"实验——同一套配置反复多跑几次,量化轮次之间天然存在多大波动,再把这个波动范围和统计置信区间结合起来,只有真实差异同时超过噪声地板和置信区间,才判定为真正的回归,并且要求参与判定的样本量不能太小。

### 3.3 重排层指标(设计已定,代码未实现)——回答"重排相对不重排,到底有没有起效"

早期设计见 [phase2_rerank_design.md](../archive/design-v1/phase2_rerank_design.md)。这一层和检索层要回答的问题本质不同:检索层问"候选里有没有标准答案",重排层问"重排这一步比不重排(即只用融合排序)相比,有没有真的把对的结果排得更靠前"。设计上定了两条关键规则:

第一,同一轮评测要在同一份候选集合上,**同时产出"重排生效后的顺序"和"降级为纯融合排序的顺序"**两个版本,而且这份候选集合必须先统一过滤掉查不到正文的项,保证两条序列的候选范围和长度完全一致,不然两者没法比。

第二,指标不能只看重排后的绝对分数,必须看**相对融合基线的增量**:

- `ΔNDCG@k = NDCG@k(重排序) − NDCG@k(纯融合序)`
- `ΔMRR = MRR(重排序) − MRR(纯融合序)`

增量大于零才说明重排真的把结果排得更靠前;接近零或小于零说明这批数据上重排没有带来收益甚至有反作用。

设计里还特别强调必须同时报告"重排实际生效的样本比例"(`rerank_applied` 率)。原因是:一旦重排因为模型调用异常或索引不可用而降级,那条样本的两个序列就是同一个,增量天然为零——如果不单独报出降级样本占多少比例,大量降级样本会把整体的增量拉平,造出一个"重排好像没什么用"的假象。所以报告要同时给出"全体样本的增量"和"只统计重排真正生效那部分样本的增量"两个数字,不能只挑一个说。

### 3.4 生成与正确性层指标(设计已定,代码未实现)——回答"答案忠不忠实、对不对"

这是 Phase 3,也是链路最后一层。早期设计见 [phase3_generation_design.md](../archive/design-v1/phase3_generation_design.md)。设计上把它拆成两组指标,分别回答"答得忠不忠实、切不切题"和"答得对不对"两个不同的问题:

第一组,**生成质量(RAG Triad,不需要标准答案)**——只要有生成出来的答案和喂给模型的上下文,不需要知道标准答案是什么,就能打分:

- **忠实度(Faithfulness)**:答案里的每一条论断,能不能在给模型看的上下文里找到依据,直接抓"看似说得头头是道、实际是模型编出来的"这类幻觉,是这一组里信号最强的指标。
- **答案相关性(Answer Relevancy)**:答案和用户提的问题在语义上贴不贴合,抓"答非所问"。
- **上下文相关性(Context Relevance/Precision)**:喂给模型的上下文里,真正对回答有用的部分占多大比例,抓"塞了一堆无关内容"。

第二组,**端到端正确性(需要标准答案)**——只有黄金测试集里带了标准答案的样本才能算,缺标准答案的样本会被直接跳过并计数,不会硬凑一个假分数:

- **上下文召回(Context Recall)**:回答这道题真正需要的信息,有多少已经出现在了喂给模型的上下文里,这一项被视为"检索这一步本身对不对"的判据,而不只是生成层的问题。
- **答案正确性(Answer Correctness/Accuracy)**:生成的答案和标准答案之间,事实层面是否一致。

这两组指标都需要引入 LLM 判官打分,因为"忠实""相关""正确"这些判断本质上是语义理解,没法用公式精确定义。设计上首版直接借用业界已经验证过的 RAGAS 框架的评测 prompt,而不是自己从零设计打分标准,后续如果发现某个指标用量很大、值得投入,再考虑替换成自研判官,但打分协议保持不变、不影响上层。这一层同样受"生成器模型、判官模型、被测对话模型两两不得同名"这条防自评偏置纪律约束,并且设计文档也坦承,这条纪律只能防止最明显的同名偏置,防不住同源 LLM 之间更隐蔽的系统性偏差,所以这组指标在合成数据集上跑出来的绝对分数天然偏乐观,只适合做同一套流程内部的相对趋势追踪,不适合当作跨系统比较的绝对结论。

## 四、黄金测试集:为什么说它"有权威性"

如果说指标公式解决的是"怎么算分",那黄金测试集解决的就是"标准答案从哪儿来、这个标准答案能不能信"。这是整个质检模块设计最精细的部分,背后有一个明确的哲学转向:人工逐条抽检的成本不可持续,所以放弃追求"完美的绝对权威尺",转而追求"稳定的相对比较尺"。这不是在回避质量问题,而是承认一个更诚实的定位——质检系统最核心的用途是"换了模型供应商、调了参数之后,系统到底有没有变差",只要测试集本身的噪声在不同配置之间保持恒定,即使绝对分数存在一定噪声,也依然能够正确判断"变好了还是变差了"这个方向性问题。

黄金测试集由三条数据来源构成,权威性也建立在这三条来源各自不同的论证逻辑上。

**第一条来源:开源中文检索数据集,权威性来自第三方官方人工标注。** 这条数据是主力,直接把开源数据集的原始文档灌入评测语料,复用数据集官方发布的"查询—相关段落"人工标注结果转换成标准答案,评测系统本身完全不参与"这道题标准答案是什么"这个判断——它只做数据格式的转换和过滤。选用的具体数据集经过授权核实:一个是约十万条真实搜索查询、近九百万段落的检索数据集,标注是二值相关(相关/不相关),因为其查询分布最贴近真实搜索场景,是检索层的主力数据;另一个是约三十万条查询的数据集,标注是四级分级相关(非常相关到不相关分四档),专门用来支撑分级版本的 NDCG 计算和重排辅助验证,但文档也诚实说明,这个数据集的标注粒度是整篇文档级别,经过切分成 chunk 之后,细粒度的排序信息会被抹平,所以对重排层只能给出近似结论,不能当作精确判据。这条数据来源刻意不选垂直行业的子集(比如电商、医疗领域专用集),因为评测的系统本身面向通用多领域场景,不存在特定客户的垂直语料,通用数据集已经足够贴合,选垂直子集反而会造成场景错配。

这条来源权威性的关键逻辑是:标准答案来自第三方权威机构发布的人工标注,评测系统自己没有掺和"什么是正确答案"这个判断过程,从根本上避免了"自己出题、自己判卷"的循环论证问题。

**第二条来源:LLM 合成真实格式文档,权威性来自"构造性真值"而非"判断性真值"。** 第一条来源虽然权威,但绕开了文档解析这一步——它本身就是纯文本段落,测不到"PDF/Word/HTML 怎么被解析清洗、怎么被切分"这条真实链路上的表现。所以设计了第二条来源专门补这个缺口,核心机制是"埋点":先构造一个个"事实单元",每个事实单元包含一句陈述和一个用来定位的"锚点短语"——这个短语要求足够独特、包含具体数字或专有名词,便于后续原样找回;再用 LLM 把这些事实单元编织成一篇结构完整的文档,渲染成真实的 PDF/Word/HTML 格式;然后让这份文档走一遍和生产完全一致的解析、切分、入库流程;再拿锚点短语去真实产出的文本块里反向定位,找到这段事实最终落在了哪个 chunk 里;最后基于这个事实反向生成一个不会泄露锚点原文的问题,标准答案就是构造时预先写好的那句陈述。

这里最关键的设计巧思在于反向定位这一步:如果简单粗暴地用"完全一致的字符串匹配"去定位锚点,PDF 解析这类会改写、丢字、重新排版的解析路径,只要出现一个全角半角字符的差异就会被判定"找不到"——而这恰恰是最需要被测试到、最容易出问题的场景,如果因为匹配太严格而把这些样本系统性地丢弃,反而会造成"最该被考验的场景反而在测试集里最少见"的偏差。所以定位逻辑设计了三个由严到松的匹配档位:完全一致、归一化后(去掉全角半角、标点、空白差异)一致、模糊相似度匹配,只有连模糊匹配都找不到的才真正判定为丢失,并按"文档格式 × 解析后端"分桶统计丢失率,用来监控哪条解析路径最容易吞字。

这条来源权威性的逻辑和第一条不同:它的标准答案不是"谁来判断这题该怎么答",而是"构造时就写死在文档里的既定事实",问题的答案是构造决定的,不存在标注者主观误判的空间。另外它还有一个附带的工程优势:因为锚点信息会被持久化保存,后续如果调整了切分策略,不需要把整个黄金集重新生成一遍,只需要对同一份冻结语料重新切分、重新用锚点匹配一次,就能自动重建新的标准答案对照关系,大幅降低了因为改切分策略而报废重做的成本。

**第三条来源:自有语料反向合成,权威性来自自动化质量门禁,而不是人工抽检。** 这条路径从已经入库的评测语料里分层抽样文本块(按数据集和文本类型分桶轮转抽样,避免样本集中在少数长文档上;相邻文本块还会被组合起来构造跨段落问题),交给 LLM 反向生成"这段内容能回答的问题及其标准答案",四种问题类型(关键词直问、同义改写、长尾细节、跨文档多跳)按比例混合生成。

这条来源没有第三方人工标注背书,也没有"构造即真值"的天然保证,所以它的权威性完全建立在一套自动化质量门禁上,用机器可执行的检验取代人工抽检,包含三个信号:

第一个信号是"异模型可答性验证"——换一个和生成问题的模型不同、通常更强的模型,只给它看标准答案对应的原文,让它独立尝试回答这道题,答不出来就自动丢弃这条样本,专门抓住"这段内容其实回答不了这个问题"的情况。

第二个信号是"第三方检索回环验证",这是防止循环论证最关键的一步设计:把生成出来的问题,交给一个和被评测系统完全无关的独立检索器(通常是纯 BM25)去检索,看标准答案对应的文本块能不能被这个独立检索器找到。之所以坚持用独立检索器而不是被评测系统自己去筛选,是因为如果用被评测系统自己筛选黄金集,凡是当前配置召不回的样本都会被系统性地踢掉,这样一来黄金测试集就会向"当前这套配置本来就擅长的场景"偏置,以后再用这套集合去评测新的模型或新的参数配置,对新方案反而不公平。检索不中的样本也不会直接丢弃,而是单独归入"难例桶",跨配置对比时必须显式说明是否包含这部分,不能悄悄混入正常样本拉低平均分。

第三个信号是"答案自洽性验证",比较生成器给出的标准答案和第一个信号里独立模型给出的答案,如果语义上分歧很大,同样判定丢弃,专门用来抓生成模型自己编造事实的情况。

这三个信号共同构成一份质量门禁报告,量化给出通过率、丢弃率、难例样本占比,用这些可复算的数字作为这批数据可信度的凭据,取代了"人工抽检若干条、感觉还行"这种模糊的质量结论。

**贯穿三条来源的一条纪律:防自评偏置的模型错配要求。** 无论哪条数据来源,只要流程里涉及多个 LLM 角色(生成问题的模型、门禁复核的模型、最终评测打分的判官模型、被评测的对话模型本身),系统会强制要求这些角色两两使用不同的模型,一旦出现同名会显式告警。但文档非常坦诚地指出了这条纪律的边界:它只能防止"同一个模型自己出题自己判卷"这种最直接的偏置,防不住"虽然模型不同,但都是 LLM、都存在某些相近的判断倾向"这种更隐蔽的系统性偏差。因此涉及 LLM 判官打分的指标(尤其是生成层的忠实度、相关性),在合成数据集上的绝对分数会偏乐观,项目里明确把这类指标的用途限定为"同一套评测流程内部的相对趋势追踪",不作为跨系统比较的绝对权威结论;相比之下,第二条来源里"构造即真值"的正确性判据(标准答案是埋进文档里的既定事实),可信度要高于依赖 LLM 判官主观打分的指标。

**权威性论证之外,还有一份实证报告值得一提。** 项目团队曾经做过一次标注可靠性的复核实验:抽查几个公开数据集在三路召回评测里的失败样本,发现相当一部分"失败"其实是标注本身有问题,而不是召回系统真的没找到。于是对医疗、电商、视频三个数据集做了池化重标——把三路召回的候选结果和原始标注合并去重形成候选池,用一个独立的大模型逐条重新判定相关性,而且只增不删(绝不移除原始标注,因为实测发现大模型在判断"这条不相关、应该删掉"这个方向上并不可靠,容易误删明显相关的内容)。结果发现医疗数据集里将近一半的查询存在漏标,修正后召回率有明显提升;电商数据集漏标比例很低,标注基本可信;视频数据集因为召回本来就已经饱和,漏标不影响结论。这次实验最大的价值在于证明了一件容易被忽视的事:评测分数低,不一定是系统真的差,也可能是评测集本身有缺陷,面对这种不确定性,报告给出的是一个可信区间(修正前的下界、重标之后的上界),而不是草率地下一个"系统不行"或者"评测集有问题"的单一结论。这也正是这套黄金集设计哲学的一个缩影:承认局限、给出边界,比宣称绝对正确更有说服力。

**把以上几条捏合起来,黄金测试集"具有权威性"的完整逻辑链是这样的:** 主力数据的标准答案来自第三方官方人工标注,评测系统不参与标注决策;合成数据的标准答案是构造时写定的既定事实,不存在标注者主观误判;自动质量门禁刻意使用独立于被测系统的检索器做回环验证,避免自己筛选自己的评测集这种循环论证;语料一次性灌入后冻结只读,任何一轮评测开跑前都会先校验数据完整性,数据对不上就直接判定无效、绝不硬凑出一个看似正常实则失真的分数;最后,项目对自身局限保持坦诚——不宣称绝对权威,承认"不同配置间噪声保持恒定"只是一个待持续验证的假设,承认同源 LLM 偏置无法根除,承认小样本分桶的结论只能定性参考不能定量下结论。这种主动暴露边界、而不是把局限藏起来的态度,恰恰是这套评测体系区别于一般"自娱自乐刷分"式评测框架的地方。

## 五、存储隔离:质检结果为什么可信,不会被生产数据污染

质检结果的可信度,除了指标和黄金集本身设计得合理,还需要底层存储不被生产环境干扰。这方面有三层隔离机制。

**向量库隔离**:评测复用生产的向量存储读写组件,但实例化时使用独立的 `eval` 前缀,并且在构造阶段就强制校验前缀,不满足直接抛异常拒绝运行,这是防止误写生产向量库的最后一道防线。文本块的唯一标识用确定性算法生成(基于数据集、文档、序号三个字段做哈希),同样的输入永远得到同样的标识,这样即使语料被重新灌入,标识也不会变化,标准答案的对照关系不会因为重新入库而失效。

**关系库隔离**:评测使用独立的 MySQL 数据库,和生产同一台服务器但完全独立建库,复用账号权限只换库名,数据库表结构的演进只走评测自己的迁移链路,和生产的数据库迁移完全隔离,绝不触碰生产的业务表。

**号段隔离**:评测使用的用户标识和数据集标识都预留在专门的号段区间内,这个标识只是一个用于向量库分桶路由的技术常量,不代表真实的用户身份或登录态,生产侧的真实业务不会占用这个号段,从根本上避免了评测流量和生产流量在路由层面产生混淆。

每一轮评测运行都会记录下当次使用的模型指纹(用的是哪个向量模型、哪个稀疏检索供应商、哪种分词模式等),配合运行质量标记(有没有分路失败、有没有零结果样本),可以清楚区分哪些运行结果足够"干净"、可以被固化为后续对比的基线,哪些运行结果因为过程中出现异常而不适合作为基准参考。

## 六、小结

这套质检模块回答了 RAG 系统质检里最容易被含糊带过的三个问题。测什么:从文档清洗、召回、重排到生成正确性,分层拆解,每层有边界清晰的质检对象,不越界也不遗漏。用什么指标测:检索、重排层用确定性的数学指标,零成本、可进 CI;生成层用 LLM 判官,但明确标注其结论只能做相对趋势追踪。标准答案哪来、能不能信:开源数据靠第三方权威标注背书,合成数据靠构造性真值和独立检索回环的自动化质量门禁背书,并且始终对自身的局限保持坦诚,把"这个数字可能不够准"这件事写进报告里,而不是藏起来。

正是这种"追求可复现、追求方法论自洽、并且主动暴露局限"的工程姿态,让这套评测体系不只是一个刷分工具,而是能够真正支撑"这次调参/换模型到底有没有把系统变好"这个业务判断的可信依据。
